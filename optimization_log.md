### 已识别问题

#### 问题 #1: JSON 解析错误
**位置**: `src/limem/ltm.py:106` - `extract_entities_from_llm()`

**现象**:
```
json.decoder.JSONDecodeError: Extra data: line 1 column 38 (char 37)
```

**原因**: LLM 返回的实体提取结果包含额外文本，不是纯 JSON 数组。例如：
```json
["实体1", "实体2"] 这里有一些额外的说明文字
```

**影响**: 导致处理某些 episode 时崩溃

**修复计划**: 增强 JSON 解析的容错性，提取有效 JSON 部分

### 2026-02-15 - 问题 #2: 事件相似度阈值过高导致记忆合并

**问题描述**:
测试通过率仅为 8.7% (2/23)，诊断发现数据库中只有 2 个事件：
1. "用户提到身体不适，车机建议多休息。" - 50 个实体
2. "用户请求播放关于长城历史的纪录片" - 28 个实体

**根本原因**:
`SIMILARITY_THRESHOLD = 0.85` 过高，导致不同的事件被错误地合并到已有事件中。从构建日志可以看到：
```
🔎 Top similarity: 1.0000 -> 用户请求播放关于长城历史的纪录片
🔍 Found existing memory: 用户请求播放关于长城历史的纪录片
```

几乎所有新事件的向量相似度都 >= 0.85，导致它们被合并而不是创建新事件。

**影响**:
- 记忆碎片化：29 个不同场景的对话被合并成 2 个事件
- 检索准确性下降：无法区分不同场景的具体事件
- 语义信息丢失：事件摘要过于泛化

**修复方案**:
降低 `SIMILARITY_THRESHOLD` 从 0.85 到 0.70，允许更细粒度的事件区分

**修改文件**: `.env`

**修改内容**:
```
SIMILARITY_THRESHOLD=0.70  # 从 0.85 降到 0.70
```

**理论依据**:
- 阈值 0.85 意味着只有完全相同的事件才会被分开
- 阈值 0.70 允许语义相关但不同的事件分开存储
- 仍能合并真正重复的事件（如重复的相同请求）

**测试计划**:
1. 删除旧数据库
2. 使用新的阈值重新构建
3. 验证事件数量增加（预期 >10 个事件）
4. 重新运行测试用例

**预期效果**:
- 事件数量从 2 个增加到 15-20 个
- 每个事件关联的实体数量减少（更精确）
- 检索准确性显著提升
