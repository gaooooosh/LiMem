# LiMem 长期记忆系统算法说明文档

## 概述

LiMem 是一个基于图数据库（Kuzu）和大语言模型（LLM）的长期记忆系统，专为车载语音助手场景设计。系统实现了**情景记忆整合**、**基于实体的检索**和**加权记忆衰减**三大核心功能。

---

## 系统架构

LiMem 采用**四阶段流水线架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                        数据输入                              │
│                 原始对话记录（Episode）                       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 1：事件与实体提取（LLM）                        │
│  • 提取结构化事件（摘要、参与者、时间、地点、动作）           │
│  • 提取相关实体（人名、地名、概念）                          │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 2：记忆整合（向量相似度）                       │
│  • 向量相似度搜索：找到最相似的已有事件                       │
│  • 合并或创建：相似度 > 阈值则合并，否则创建新事件            │
│  • 关系更新：更新 Event-Entity 关系权重                       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│              图数据库存储（Kuzu）                             │
│  • Event 节点：整合后的语义记忆                              │
│  • Entity 节点：符号节点（人物、地点、概念）                  │
│  • INVOLVES 关系：加权关系（含时间戳和验证计数）              │
└─────────────────────────────────────────────────────────────┘
```

## 存储阶段算法详解

### 1. 事件提取（两阶段 LLM 调用）

**设计理念**：将事件提取和实体提取分离，避免上下文溢出，允许独立优化。

#### 1.1 事件提取
- **输入**：原始对话文本（episode_content）
- **输出**：结构化事件对象
  - `summary`：事件摘要（关键细节：人名、地名、具体内容）
  - `participants`：参与者列表
  - `time_range`：时间范围（start, end, display_time_bucket）
  - `location`：地点信息
  - `action`：动作描述
  - `causality`：因果关系
  - `evidence`：证据片段

**Prompt 优化要点**（来自调试记录）：
- 要求摘要必须包含关键细节（歌曲名、歌手名、具体数值等）
- 避免过度简化导致语义信息丢失

#### 1.2 实体提取
- **输入**：原始对话文本
- **输出**：实体列表（人名、地名、概念、媒体名称等）

### 2. 记忆整合机制

#### 2.1 向量相似度搜索
```python
# 生成事件摘要的嵌入向量
embedding = get_embedding(summary)

# 在已有事件中搜索最相似的
best_id, best_summary, similarity = _find_most_similar_event(embedding)
```

**关键参数**：
- `SIMILARITY_THRESHOLD = 0.70`（优化后）
  - 原始值 0.85 导致过度合并（29个对话 → 2个事件）
  - 优化后允许语义相关但不同的事件分开存储

#### 2.2 合并或创建决策
```python
if similarity > SIMILARITY_THRESHOLD:
    # 合并到已有事件
    event_id = best_id
    merged_evidence = merge_evidence(existing, incoming)
    update_event_embedding(event_id, embedding)
    update_last_active(event_id, current_time)
else:
    # 创建新事件
    event_id = create_new_event(summary, embedding, ...)
```

#### 2.3 关系权重更新
对每个提取的实体：
1. 确保实体节点存在（不存在则创建 + 生成嵌入向量）
2. 创建或更新 `INVOLVES` 关系（Event → Entity）
3. 递增 `c_valid`（验证计数）

```python
# 创建/更新 INVOLVES 关系
MATCH (e:Event {id: $event_id}), (ent:Entity {id: $entity_id})
MERGE (e)-[r:INVOLVES]->(ent)
SET r.c_valid = COALESCE(r.c_valid, 0) + 1,
    r.t_created = COALESCE(r.t_created, $current_time),
    r.t_valid = $current_time
```

### 3. 权重计算公式

**核心公式**：
```
weight = log(1 + c_valid) * exp(-DECAY_RATE * time_diff)
```

**参数说明**：
- `c_valid`：验证计数（关系被确认的次数）
- `DECAY_RATE = 1e-8`：衰减率
  - 半衰期 ≈ 2.2 年（允许记忆长期保留）
  - 公式：`half_life = ln(2) / DECAY_RATE`
- `time_diff = t_now - t_valid`：时间差（秒）

**Bug 修复历史**（来自 tuning_log）：
1. ❌ **错误公式**：`exp(-lambda * log(1 + abs(time_diff)))` → 所有事件权重相同
2. ❌ **错误参考时间**：使用 `max(t_valid)` 作为参考时间 → 最新事件时间差为 0
3. ✅ **正确公式**：使用线性时间差 + 指数衰减 + 当前时间戳

## 检索阶段算法详解

检索采用**四阶段流水线**：

```
┌─────────────────────────────────────────────────────────────┐
│          阶段 1：实体提取（LLM）                             │
│  从自然语言查询中提取关键实体                                │
│  例："放孩子最喜欢的那个动画片" → ["孩子", "最喜欢", "动画片"]│
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│        阶段 2：图路径搜索（混合实体匹配）                     │
│  • 精确匹配：Cypher 查询匹配实体名称（权重 = 1.0）            │
│  • 向量匹配：实体嵌入相似度匹配（权重 = similarity²）         │
│  • 合并结果：追踪 match_type ("exact"/"vector"/"both")       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 3：权重重排序                                  │
│  计算每个事件的综合权重并排序                                │
│  weight = log(1 + c_valid) * exp(-DECAY_RATE * time_diff)   │
│           * (Σ entity_match_weights)²                       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 4：LLM 总结                                    │
│  基于 Top-K 事件生成自然语言回答                             │
└─────────────────────────────────────────────────────────────┘
```

### 1. 实体提取

**输入**：自然语言查询
**输出**：实体列表

**过滤规则**：
- 移除停用词（的、了、吗、呢、啊等）
- 移除无意义的单字（除非是数字或英文字母）
- 保留有语义的单字（如"25"、"K"）

### 2. 混合实体匹配

#### 2.1 精确匹配
```cypher
MATCH (e:Event)-[:INVOLVES]->(ent:Entity)
WHERE ent.id IN $entity_list
RETURN e.*, ent.id as matched_entity
```
- 匹配权重：`1.0`（完全匹配）

#### 2.2 向量匹配
```python
# 为查询实体生成嵌入向量
query_embedding = get_embedding(entity)

# 在数据库中搜索相似实体
similar_entities = vector_search(query_embedding, top_k=10, threshold=0.5)

# 匹配权重 = 相似度²
entity_match_weight = similarity ** 2
```

**关键参数**：
- `SEARCH_VECTOR_THRESHOLD = 0.5`：向量相似度阈值
- `SEARCH_VECTOR_TOP_K = 10`：每个实体最多返回 10 个相似实体

#### 2.3 结果合并
- 同一事件可能来自精确匹配和向量匹配
- 记录 `match_type`：`"exact"` / `"vector"` / `"both"`
- 合并 `entity_match_weights` 字典

### 3. 权重重排序

**完整权重公式**：
```python
weight = log(1 + c_valid) * exp(-DECAY_RATE * time_diff) * (Σ entity_match_weights)²
```

**硬过滤条件**（权重 = 0）：
- `t_expired is not None`：事件已过期
- `t_invalid is not None and t_now >= t_invalid`：事件已失效
- `time_diff < 0`：未来事件（t_valid > t_now）

**实体匹配放大**：
- 使用 `(Σ entity_match_weights)²` 放大语义相关性
- 确保语义相关的事件排名更高，即使时间较旧

**参考时间修复**：
```python
# ❌ 错误：使用最大事件时间
t_now = max(max(t_valid), max(last_active))

# ✅ 正确：使用当前时间戳
t_now = int(time.time())
```

### 4. LLM 总结

**输入**：
- 用户查询（query）
- Top-K 事件列表（event summaries）

**输出**：自然语言回答

## 关键参数配置

### 存储阶段参数

| 参数名 | 默认值 | 说明 | 调优建议 |
|--------|--------|------|----------|
| `SIMILARITY_THRESHOLD` | 0.70 | 事件相似度阈值 | 过高导致过度合并，过低导致碎片化 |
| `DECAY_RATE` | 1e-8 | 时间衰减率 | 半衰期 ≈ 2.2 年，适合长期记忆 |
| `EPISODE_TTL` | 3600 | Episode 节点生存时间（秒） | 1 小时后自动清理原始对话 |
| `PRUNE_C_VALID_THRESHOLD` | 100 | 永久特征提升阈值 | c_valid > 100 提升为 PERMANENT_TRAIT |

### 检索阶段参数

| 参数名 | 默认值 | 说明 | 调优建议 |
|--------|--------|------|----------|
| `SEARCH_TOP_K` | 5 | 返回的 Top-K 事件数量 | 根据上下文窗口调整 |
| `SEARCH_MAX_ENTITIES` | 10 | 从查询中提取的最大实体数 | 过多会增加噪音 |
| `SEARCH_ENABLE_VECTOR_MATCH` | True | 启用向量匹配 | 提高召回率，但增加计算成本 |
| `SEARCH_VECTOR_THRESHOLD` | 0.5 | 向量相似度阈值 | 过低会引入噪音实体 |
| `SEARCH_VECTOR_TOP_K` | 10 | 每个实体返回的相似实体数 | 平衡召回率和性能 |

## 数据模型

### 节点类型

#### Event（事件节点）
```cypher
CREATE (:Event {
    id: String,              // 事件 ID（基于摘要哈希）
    summary: String,         // 事件摘要
    participants: JSON,      // 参与者列表
    time_range: JSON,        // 时间范围 {start, end, display_time_bucket}
    location: JSON,          // 地点信息
    action: String,          // 动作描述
    causality: String,       // 因果关系
    evidence: JSON,          // 证据片段列表
    consistency: Float,      // 一致性分数
    last_active: Int,        // 最后活跃时间戳
    embedding: Float[]       // 事件摘要的嵌入向量
})
```

#### Entity（实体节点）
```cypher
CREATE (:Entity {
    id: String,              // 实体名称
    type: String,            // 实体类型（UNKNOWN/PERSON/LOCATION等）
    embedding: Float[]       // 实体名称的嵌入向量
})
```

#### Episode（情景节点）
```cypher
CREATE (:Episode {
    id: String,              // Episode ID（UUID）
    content: String,         // 原始对话内容
    timestamp: Int           // 创建时间戳
})
```

### 关系类型

#### INVOLVES（Event → Entity）
```cypher
CREATE (e:Event)-[:INVOLVES {
    c_valid: Int,            // 验证计数
    t_created: Int,          // 关系创建时间
    t_valid: Int,            // 最后验证时间
    t_expired: Int,          // 过期时间（可选）
    t_invalid: Int           // 失效时间（可选）
}]->(ent:Entity)
```

#### EXTRACTED_FROM（Event → Episode）
```cypher
CREATE (e:Event)-[:EXTRACTED_FROM]->(ep:Episode)
```

## 调试与优化历程

### 问题 1：JSON 解析错误
- **现象**：LLM 返回包含额外文本的 JSON
- **解决**：实现 `robust_json_loads()` 函数，提取有效 JSON 部分

### 问题 2：事件过度合并
- **现象**：29 个对话被合并成 2 个事件
- **原因**：`SIMILARITY_THRESHOLD = 0.85` 过高
- **解决**：降低到 0.70，事件数量增加到 22 个
- **结果**：检索准确率从 8.7% 提升到 82.6%

### 问题 3：时间权重无区分度
- **现象**：所有事件权重相同（0.693147）
- **原因**：
  1. 公式错误：使用 `log(1 + abs(time_diff))` 而非线性时间差
  2. 参考时间错误：使用 `max(t_valid)` 而非当前时间
- **解决**：
  1. 使用线性时间差：`time_diff = t_now - t_valid`
  2. 使用当前时间戳：`t_now = int(time.time())`
  3. 正确的指数衰减：`exp(-DECAY_RATE * time_diff)`

### 问题 4：事件摘要丢失细节
- **现象**：无法检索到歌曲名"还是分开"、歌手"张叶蕾"
- **原因**：Prompt 要求"简短描述"导致关键细节丢失
- **解决**：优化 Prompt，要求摘要必须包含人名、地名、具体内容、数据等

## 总结

LiMem 系统通过以下核心机制实现长期记忆管理：

1. **双层提取**：分离事件和实体提取，避免上下文溢出
2. **智能整合**：基于向量相似度的记忆合并，避免重复存储
3. **加权衰减**：时间衰减 + 频率强化 + 实体匹配的多维权重
4. **混合检索**：精确匹配 + 向量匹配的双重召回机制
5. **语义总结**：LLM 生成自然语言回答

系统已达到 87% 的检索准确率，能够有效支持车载语音助手的长期记忆需求。
