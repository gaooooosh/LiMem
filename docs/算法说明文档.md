# LiMem 长期记忆系统算法说明文档

## 概述

LiMem 是一个基于图数据库（Kuzu）和大语言模型（LLM）的长期记忆系统，专为车载语音助手场景设计。系统实现了**情景记忆整合**、**基于实体的检索**和**加权记忆衰减**三大核心功能。

---

## 系统架构

LiMem 采用**四阶段流水线架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                        数据输入                              │
│                 原始对话记录（Episode）                       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 1：事件与实体提取（LLM）                        │
│  • 提取结构化事件（摘要、参与者、时间、地点、动作）           │
│  • 提取相关实体（人名、地名、概念）                          │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 2：记忆整合（向量相似度）                       │
│  • 向量相似度搜索：找到最相似的已有事件                       │
│  • 合并或创建：相似度 > 阈值则合并，否则创建新事件            │
│  • 关系更新：更新 Event-Entity 关系权重                       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│              图数据库存储（Kuzu）                             │
│  • Event 节点：整合后的语义记忆                              │
│  • Entity 节点：符号节点（人物、地点、概念）                  │
│  • INVOLVES 关系：加权关系（含时间戳和验证计数）              │
└─────────────────────────────────────────────────────────────┘
```

## 存储阶段算法详解

### 1. 事件提取（两阶段 LLM 调用）

**设计理念**：将事件提取和实体提取分离，避免上下文溢出，允许独立优化。

#### 1.1 事件提取
- **输入**：原始对话文本（episode_content）
- **输出**：结构化事件对象
  - `summary`：事件摘要（关键细节：人名、地名、具体内容）
  - `participants`：参与者列表
  - `time_range`：时间范围（start, end, display_time_bucket）
  - `location`：地点信息
  - `action`：动作描述
  - `causality`：因果关系
  - `evidence`：证据片段

**Prompt 优化要点**（来自调试记录）：
- 要求摘要必须包含关键细节（歌曲名、歌手名、具体数值等）
- 避免过度简化导致语义信息丢失

#### 1.2 实体提取
- **输入**：原始对话文本
- **输出**：实体列表（人名、地名、概念、媒体名称等）

### 2. 记忆整合机制

#### 2.1 多维度相似度搜索（核心算法）

系统采用**四维度加权相似度算法**，综合考虑语义、实体、时间和动作四个维度，避免单一向量相似度的局限性。

**算法公式**：
```python
综合相似度 = α × 语义相似度 (Semantic)  # 向量嵌入的余弦相似度
           + β × 实体重叠度 (Entity)  # Jaccard 相似度
           + γ × 时间接近度 (Time)    # 时间差指数衰减
           + δ × 动作匹配度 (Action)  # (0 或 1)：完全匹配

其中：
- α = MERGE_WEIGHT_SEMANTIC = 0.4
- β = MERGE_WEIGHT_ENTITY = 0.3
- γ = MERGE_WEIGHT_TIME = 0.2
- δ = MERGE_WEIGHT_ACTION = 0.1
- α + β + γ + δ = 1.0
```

**实现代码**：
```python
def _find_most_similar_event(self, embedding, current_time, incoming_entities, incoming_action):
    """
    多维度事件相似度搜索

    返回: (best_id, best_summary, combined_score, debug_info)
    """
    # 1. 语义相似度 (0-1)：向量嵌入的余弦相似度
    semantic_sim = cosine_similarity(embedding, stored_embedding)

    # 2. 实体重叠度 (0-1)：Jaccard 相似度
    incoming_set = set(entity['name'] for entity in incoming_entities)
    stored_set = set(stored_entities)
    intersection = len(incoming_set & stored_set)
    union = len(incoming_set | stored_set)
    entity_sim = intersection / union if union > 0 else 0.0

    # 3. 时间接近度 (0-1)：指数衰减
    time_diff = abs(current_time - last_active)
    if time_diff <= MERGE_TIME_WINDOW:  # 默认 300 秒（5 分钟）
        time_sim = exp(-3.0 * time_diff / MERGE_TIME_WINDOW)
    else:
        time_sim = 0.0

    # 4. 动作匹配度 (0 或 1)：完全匹配
    action_sim = 1.0 if incoming_action == stored_action else 0.0

    # 加权组合
    combined_score = (
        MERGE_WEIGHT_SEMANTIC * semantic_sim +
        MERGE_WEIGHT_ENTITY * entity_sim +
        MERGE_WEIGHT_TIME * time_sim +
        MERGE_WEIGHT_ACTION * action_sim
    )

    return combined_score
```

**关键参数**：
- `SIMILARITY_THRESHOLD = 0.65`（多维度综合阈值）
  - 注意：这是综合评分，比纯向量相似度阈值更低
  - 推荐范围：0.60-0.75

- `MERGE_TIME_WINDOW = 300`（时间窗口，秒）
  - 时间窗口内的事件获得更高的时间相似度
  - 推荐范围：300-900 秒（5-15 分钟）

**算法优势**：
1. **鲁棒性**：即使 LLM 生成的摘要表述不同，也能通过实体、时间、动作识别相似事件
2. **可解释性**：每个维度的得分都有明确的物理意义
3. **可调节性**：通过权重参数可以针对不同场景优化

**案例对比**：

| 场景 | 旧算法（纯向量） | 新算法（多维度） | 改进 |
|------|-----------------|-----------------|------|
| 音乐播放事件 | 两个相似事件被分开 | 成功合并为一个事件 | ✅ 减少碎片化 |
| 相同时间不同事件 | 可能错误合并 | 时间接近但实体不同 → 分开 | ✅ 提高精确度 |
| 相同动作不同对象 | 可能合并 | 动作相同但实体不同 → 分开 | ✅ 语义更准确 |

#### 2.2 合并或创建决策
```python
if combined_score > SIMILARITY_THRESHOLD:
    # 合并到已有事件
    event_id = best_id
    merged_evidence = merge_evidence(existing, incoming)
    update_event_embedding(event_id, embedding)
    update_last_active(event_id, current_time)
    print(f"🔍 Found existing memory: {best_summary}")
else:
    # 创建新事件
    event_id = create_new_event(summary, embedding, ...)
    print("🆕 Creating new memory...")
```

**调试输出示例**：
```
🔎 Top match (combined=0.9367)
   └─ semantic=0.992, entity=0.800, time=1.000, action=1.000
🔍 Found existing memory: 用户在2025-12-28下午4点左右请求播放首音乐...
```

#### 2.3 关系权重更新
对每个提取的实体：
1. 确保实体节点存在（不存在则创建 + 生成嵌入向量）
2. 创建或更新 `INVOLVES` 关系（Event → Entity）
3. 递增 `c_valid`（验证计数）

```python
# 创建/更新 INVOLVES 关系
MATCH (e:Event {id: $event_id}), (ent:Entity {id: $entity_id})
MERGE (e)-[r:INVOLVES]->(ent)
SET r.c_valid = COALESCE(r.c_valid, 0) + 1,
    r.t_created = COALESCE(r.t_created, $current_time),
    r.t_valid = $current_time
```

### 3. 权重计算公式

**核心公式**：
```
weight = log(1 + c_valid) * exp(-DECAY_RATE * time_diff)
```

**参数说明**：
- `c_valid`：验证计数（关系被确认的次数）
- `DECAY_RATE = 1e-8`：衰减率
  - 半衰期 ≈ 2.2 年（允许记忆长期保留）
  - 公式：`half_life = ln(2) / DECAY_RATE`
- `time_diff = t_now - t_valid`：时间差（秒）

**Bug 修复历史**（来自 tuning_log）：
1. ❌ **错误公式**：`exp(-lambda * log(1 + abs(time_diff)))` → 所有事件权重相同
2. ❌ **错误参考时间**：使用 `max(t_valid)` 作为参考时间 → 最新事件时间差为 0
3. ✅ **正确公式**：使用线性时间差 + 指数衰减 + 当前时间戳

## 检索阶段算法详解

检索采用**四阶段流水线**：

```
┌─────────────────────────────────────────────────────────────┐
│          阶段 1：实体提取（LLM）                             │
│  从自然语言查询中提取关键实体                                │
│  例："放孩子最喜欢的那个动画片" → ["孩子", "最喜欢", "动画片"]│
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│        阶段 2：图路径搜索（混合实体匹配）                     │
│  • 精确匹配：Cypher 查询匹配实体名称（权重 = 1.0）            │
│  • 向量匹配：实体嵌入相似度匹配（权重 = similarity²）         │
│  • 合并结果：追踪 match_type ("exact"/"vector"/"both")       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 3：权重重排序                                  │
│  计算每个事件的综合权重并排序                                │
│  weight = log(1 + c_valid) * exp(-DECAY_RATE * time_diff)   │
│           * (Σ entity_match_weights)²                       │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│          阶段 4：LLM 总结                                    │
│  基于 Top-K 事件生成自然语言回答                             │
└─────────────────────────────────────────────────────────────┘
```

### 1. 实体提取

**输入**：自然语言查询
**输出**：实体列表

**过滤规则**：
- 移除停用词（的、了、吗、呢、啊等）
- 移除无意义的单字（除非是数字或英文字母）
- 保留有语义的单字（如"25"、"K"）

### 2. 混合实体匹配（分层检索策略）

系统采用**分层混合匹配算法**，结合精确字符串匹配和向量语义匹配，提高检索召回率。

#### 2.1 第一层：精确匹配（Exact Match）
```cypher
-- Cypher 查询：精确字符串匹配
MATCH (e:Event)-[r:INVOLVES]->(ent:Entity)
WHERE ent.id IN $entity_list
RETURN e.id, e.summary, e.last_active, r.c_valid, r.t_expired, r.t_invalid,
       collect(ent.id) as matched_entities
```

**特点**：
- 匹配权重：`1.0`（完全匹配）
- 速度快：直接利用数据库索引
- 精确度高：零误报率
- 标记类型：`match_type = "exact"`

**示例**：
```
查询实体: ["长城", "北京"]
匹配结果: Event1 (包含实体: 长城, 北京, 用户)
         Event2 (包含实体: 长城, 纪录片)
```

#### 2.2 第二层：向量匹配（Vector Match）
```python
# 为查询实体生成嵌入向量
query_embedding = get_embedding(entity_name)

# 在数据库中搜索相似实体（Python 实现）
resp = conn.execute("MATCH (ent:Entity) RETURN ent.id, ent.embedding")
similar_entities = []

while resp.has_next():
    entity_id, stored_embedding = resp.get_next()
    similarity = cosine_similarity(query_embedding, stored_embedding)

    if similarity >= SEARCH_VECTOR_THRESHOLD:  # 默认 0.5
        similar_entities.append({
            'entity_id': entity_id,
            'similarity': similarity,
            'match_weight': similarity ** 2  # 平方放大差异
        })

# 按 Top-K 过滤
similar_entities = sorted(similar_entities, key=lambda x: x['similarity'], reverse=True)
similar_entities = similar_entities[:SEARCH_VECTOR_TOP_K]  # 默认 10
```

**特点**：
- 匹配权重：`similarity²`（平方放大语义相关性）
- 召回率高：能匹配语义相似但表述不同的实体
- 标记类型：`match_type = "vector"`

**示例**：
```
查询实体: "医院"
向量匹配: "北京协和医院" (similarity=0.82, weight=0.67)
         "朝阳医院" (similarity=0.78, weight=0.61)
         "诊所" (similarity=0.65, weight=0.42)
```

#### 2.3 结果合并与去重
```python
# 合并两层结果
event_map = {}  # event_id -> event_data

# 处理精确匹配
for event in exact_match_results:
    event_map[event.id] = {
        'match_type': 'exact',
        'entity_match_weights': {entity: 1.0 for entity in event.matched_entities}
    }

# 处理向量匹配
for event in vector_match_results:
    if event.id in event_map:
        # 已存在：更新为 "both" 类型
        event_map[event.id]['match_type'] = 'both'
        event_map[event.id]['entity_match_weights'].update(event.entity_weights)
    else:
        # 新事件：添加
        event_map[event.id] = {
            'match_type': 'vector',
            'entity_match_weights': event.entity_weights
        }
```

**match_type 含义**：
- `"exact"`：仅通过精确匹配找到
- `"vector"`：仅通过向量匹配找到
- `"both"`：同时通过精确和向量匹配找到（置信度最高）

#### 2.4 实体匹配权重计算
```python
# 实体匹配权重 = 精确匹配(1.0) 或 向量相似度²
entity_match_weight = 1.0 if match_type == 'exact' else similarity ** 2

# 事件的实体匹配总分 = Σ(实体匹配权重)²
event_entity_score = sum(entity_match_weights.values()) ** 2
```

**平方放大的目的**：
- 放大语义相关性，确保语义相关的事件排名更高
- 即使时间较旧，如果实体匹配度高，仍然能排在前面

**关键参数**：
- `SEARCH_ENABLE_VECTOR_MATCH = True`：启用向量匹配
- `SEARCH_VECTOR_THRESHOLD = 0.5`：向量相似度阈值
- `SEARCH_VECTOR_TOP_K = 10`：每个实体返回的相似实体数

**性能优化**：
- 实体嵌入向量在创建时生成并存储
- 向量搜索在 Python 层实现（适合中小规模数据）
- 未来可迁移到数据库原生向量索引（如 Kuzu 0.4.0+ 的向量索引功能）

### 3. 权重重排序

**完整权重公式**：
```python
weight = log(1 + c_valid) * exp(-DECAY_RATE * time_diff) * (Σ entity_match_weights)²
```

**硬过滤条件**（权重 = 0）：
- `t_expired is not None`：事件已过期
- `t_invalid is not None and t_now >= t_invalid`：事件已失效
- `time_diff < 0`：未来事件（t_valid > t_now）

**实体匹配放大**：
- 使用 `(Σ entity_match_weights)²` 放大语义相关性
- 确保语义相关的事件排名更高，即使时间较旧

**参考时间修复**：
```python
# ❌ 错误：使用最大事件时间
t_now = max(max(t_valid), max(last_active))

# ✅ 正确：使用当前时间戳
t_now = int(time.time())
```

### 4. LLM 总结

**输入**：
- 用户查询（query）
- Top-K 事件列表（event summaries）

**输出**：自然语言回答

## 关键参数配置

### 存储阶段参数

#### 事件合并参数（多维度）

| 参数名 | 默认值 | 说明 | 调优建议 |
|--------|--------|------|----------|
| `SIMILARITY_THRESHOLD` | **0.65** | 多维度综合相似度阈值 | 0.60-0.75（比纯向量阈值更低） |

- 实体驱动场景：`SEMANTIC=0.3, ENTITY=0.5, TIME=0.1, ACTION=0.1`
- 时间敏感场景：`SEMANTIC=0.3, ENTITY=0.2, TIME=0.4, ACTION=0.1`

| 参数名 | 默认值 | 说明 | 调优建议 |

| `PRUNE_C_VALID_THRESHOLD` | 100 | 永久特征提升阈值 | c_valid > 100 提升为 PERMANENT_TRAIT |

| 参数名 | 默认值 | 说明 | 调优建议 |
|--------|--------|------|----------|

### 节点类型

```cypher
CREATE (:Event {
    id: String,              // 事件 ID（基于摘要哈希）

```

```cypher
CREATE (:Entity {
    id: String,              // 实体名称

})
```

```cypher
CREATE (:Episode {
    id: String,              // Episode ID（UUID）
    content: String,         // 原始对话内容
    timestamp: Int           // 创建时间戳
})
```

#### INVOLVES（Event → Entity）
```cypher
CREATE (e:Event)-[:INVOLVES {

    t_valid: Int,            // 最后验证时间

```

```cypher
CREATE (e:Event)-[:EXTRACTED_FROM]->(ep:Episode)
```

## 调试与优化历程

### 问题 1：JSON 解析错误
- **现象**：LLM 返回包含额外文本的 JSON
- **解决**：实现 `robust_json_loads()` 函数，提取有效 JSON 部分

### 问题 2：事件过度合并
- **现象**：29 个对话被合并成 2 个事件
- **原因**：`SIMILARITY_THRESHOLD = 0.85` 过高
- **解决**：降低到 0.70，事件数量增加到 22 个
- **结果**：检索准确率从 8.7% 提升到 82.6%

### 问题 3：时间权重无区分度
- **现象**：所有事件权重相同（0.693147）
- **原因**：
  1. 公式错误：使用 `log(1 + abs(time_diff))` 而非线性时间差
  2. 参考时间错误：使用 `max(t_valid)` 而非当前时间
- **解决**：
  1. 使用线性时间差：`time_diff = t_now - t_valid`
  2. 使用当前时间戳：`t_now = int(time.time())`
  3. 正确的指数衰减：`exp(-DECAY_RATE * time_diff)`

### 问题 4：事件摘要丢失细节
- **现象**：无法检索到歌曲名"还是分开"、歌手"张叶蕾"
- **原因**：Prompt 要求"简短描述"导致关键细节丢失
- **解决**：优化 Prompt，要求摘要必须包含人名、地名、具体内容、数据等
