## Part 1: 项目背景与目标

---

### 第1页: 问题背景

**标题**: 为什么需要长时记忆？

**要点**:
- 当前语音助手的局限：会话结束后记忆丢失
- 用户期望：系统能"记住"我的偏好、习惯、历史交互
- 车载场景特点：高频使用、多用户、跨天跨周交互

**核心问题**:
> "上周我说过喜欢什么音乐？"
> "我上次去医院是什么时候？"

**演讲备注**:
- 举例：用户周一提到要去医院，周三问"我那天说要去哪来着"
- 现有系统无法回答，因为会话已结束

---

### 第2页: 项目目标

**标题**: LiMem 的设计目标

**要点**:
1. **记忆持久化**: 跨会话保留用户交互信息
2. **智能压缩**: 从原始对话提取结构化记忆
3. **自动遗忘**: 模拟人类记忆衰减机制
4. **精准检索**: 基于语义和时间的混合检索

**关键指标**:
- 记忆保留周期：年级别（半衰期 ~2.2年）
- 检索响应：< 1秒
- 事件合并准确率：目标 > 85%

---

### 第3页: 应用场景

**标题**: 车载语音助手场景

**典型场景**:

| 场景 | 用户问题 | 系统响应 |
|------|----------|----------|
| 偏好记忆 | "我平时早上听什么？" | 基于历史播放记录回答 |
| 行程记忆 | "我上次去长城是什么时候？" | 检索导航记录 |
| 健康关怀 | "我最近说过身体不舒服吗？" | 回溯相关对话 |
| 知识积累 | "端午节习俗我们聊过什么？" | 提取历史问答内容 |

### 第7页: 事件提取算法

**标题**: 两阶段 LLM 事件提取

**设计动机**:
- 避免 LLM 上下文溢出
- 允许独立优化各阶段 Prompt

**流程**:
```
Episode (原始对话文本)
        │
        ├──▶ Stage 1: extract_event_from_llm()
        │    输出: {
        │      summary: "用户想看长城纪录片",
        │      action: "播放视频",
        │      causality: "用户对历史感兴趣",
        │      time_range: {...},
        │      location: {...},
        │      participants: [...],
        │      evidence: [...]
        │    }
        │
        └──▶ Stage 2: extract_entities_from_llm()
             输出: ["长城", "纪录片", "腾讯视频", "历史"]
```

---

### 第8页: 多维度事件合并

**标题**: 相似事件合并算法

**问题**: 如何判断两个事件是否应该合并？

**解决方案**: 四维相似度融合

```python
combined_score = (
    0.4 × semantic_sim +    # 语义相似度（向量余弦）
    0.3 × entity_sim +      # 实体重叠（Jaccard）
    0.2 × time_sim +        # 时间接近度（指数衰减）
    0.1 × action_sim        # 动作匹配（精确匹配）
)

if combined_score > 0.65:  # SIMILARITY_THRESHOLD
    合并到已有事件
else:
    创建新事件
```

**示例**:
- Episode A: "播放长城纪录片"
- Episode B: "长城有多长？"
- → 语义相似 + 共享实体"长城" + 时间接近 → **合并**

---

### 第9页: 权重计算与衰减

**标题**: 记忆权重衰减机制

**核心公式**:
```
weight = log(1 + c_valid) × exp(-DECAY_RATE × time_diff)
```

**参数说明**:
- `c_valid`: 事件被确认/重复的次数
- `time_diff`: 距离上次激活的时间（秒）
- `DECAY_RATE = 1e-8`: 衰减率

**衰减曲线**:

| 时间 | 相对权重 | 说明 |
|------|----------|------|
| 0 天 | 100% | 刚发生 |
| 81 天 | ~50% | DECAY_RATE=1e-7 |
| 2.2 年 | ~50% | DECAY_RATE=1e-8（默认）|
| 8 天 | ~50% | DECAY_RATE=1e-6 |

**设计理念**: 模拟人类记忆——频繁回忆的事情记得久

---

### 第10页: 混合实体匹配

**标题**: 三层实体匹配策略

**检索时的实体匹配**:

| 匹配类型 | 条件 | 权重 | 示例 |
|----------|------|------|------|
| 精确匹配 | query == db | 1.0 | "长城" == "长城" |
| 包含匹配 | query in db | 0.9 | "医院" in "北京协和医院" |
| 向量匹配 | cosine > 0.5 | sim² | "长城" ≈ "万里长城" |

**匹配因子计算**:
```
if 有精确匹配:
    entity_factor = max(weight) × (1 + 0.5 × (count-1))
else:
    entity_factor = sum(fuzzy_weights) × 0.5  # 降权
```

**设计目的**: 精确匹配优先，避免模糊匹配淹没结果

## Part 6: 待完善工作

---

### 第14页: 事件提取优化（核心痛点）

**标题**: 当前问题：事件粒度过细

**现状**:
```
Episode : Event = 1 : 1  (每个对话片段一个事件)
```

**问题**:
1. 相关对话未融合（如 5 次提到"长城"产生 5 个事件）
2. Event 字段未充分利用（causality, consistency 等）
3. 缺乏跨会话的事件整合

**目标**:
```
多个相关 Episode → 融合为 1 个抽象 Event
```

**示例**:
- Episode 1: "播放长城纪录片"
- Episode 2: "长城有多长？"
- Episode 3: "我想去长城看看"
- → 融合为 Event: "用户对长城有浓厚兴趣"
