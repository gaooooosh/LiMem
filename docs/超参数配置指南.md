# LiMem 超参数配置指南

## 一、核心超参数总览表

### 1. 多维度事件合并参数（存储阶段）

| 参数名 | 推荐值 | 可调范围 | 含义 | 实际影响 | 维度类型 |
|--------|--------|----------|------|----------|----------|
| `SIMILARITY_THRESHOLD` | **0.65** | 0.60-0.75 | 多维度综合相似度阈值 | 0.85→过度合并（旧算法）<br>0.65→平衡合并与区分（新算法） | 综合评分 |
| `MERGE_WEIGHT_SEMANTIC` | **0.4** | 0.3-0.5 | 语义相似度权重 | 文本嵌入的余弦相似度 | 语义维度 |
| `MERGE_WEIGHT_ENTITY` | **0.3** | 0.2-0.4 | 实体重叠度权重 | Jaccard 相似度（共同实体比例） | 实体维度 |
| `MERGE_WEIGHT_TIME` | **0.2** | 0.1-0.3 | 时间接近度权重 | 指数衰减（越近得分越高） | 时间维度 |
| `MERGE_WEIGHT_ACTION` | **0.1** | 0.05-0.2 | 动作匹配度权重 | 完全匹配=1，否则=0 | 动作维度 |
| `MERGE_TIME_WINDOW` | **300** | 300-900 | 时间窗口（秒） | 窗口内事件时间得分衰减较慢 | 时间窗口 |

**重要说明**：
- ✅ 四个权重之和必须为 1.0
- ✅ `SIMILARITY_THRESHOLD` 是多维度综合评分，比纯向量阈值更低
- ✅ 新算法（0.65）替换旧算法（0.70），因为多维度评分通常低于纯向量相似度

### 2. 记忆衰减参数（存储阶段）

| 参数名 | 推荐值 | 可调范围 | 含义 | 实际影响 | 记忆类型 |
|--------|--------|----------|------|----------|----------|
| `DECAY_RATE` | **1e-8** | 1e-9 ~ 1e-6 | 时间衰减率，控制记忆淡化速度 | 半衰期 ≈ 2.2 年<br>1e-7 ≈ 81 天 | 长期记忆 |
| `EPISODE_TTL` | **3600** | 1800-7200 | 原始对话保留时间（秒） | 1 小时后清理<br>释放存储空间 | 情景记忆 |
| `PRUNE_C_VALID_THRESHOLD` | **100** | 50-200 | 永久特征提升阈值 | c_valid > 100 → 永久特征 | 永久记忆 |

### 2. 检索参数（搜索阶段）

| 参数名 | 推荐值 | 可调范围 | 含义 | 实际影响 | 应用场景 |
|--------|--------|----------|------|----------|----------|
| `SEARCH_TOP_K` | **5** | 3-10 | 返回的 Top-K 事件数量 | 5→平衡召回与精度<br>10→高召回但噪音多 | 答案生成 |
| `SEARCH_MAX_ENTITIES` | **10** | 5-15 | 从查询提取的最大实体数 | 10→覆盖关键实体<br>15→噪音增加 | 查询理解 |
| `SEARCH_VECTOR_THRESHOLD` | **0.5** | 0.4-0.7 | 向量相似度阈值 | 0.5→平衡召回与精度<br>0.7→高精度低召回 | 实体匹配 |
| `SEARCH_VECTOR_TOP_K` | **10** | 5-20 | 每个实体返回的相似实体数 | 10→平衡性能与召回<br>20→高召回但慢 | 向量检索 |

### 3. 混合匹配参数

| 参数名 | 推荐值 | 含义 | 权重计算 |
|--------|--------|------|----------|
| `SEARCH_ENABLE_VECTOR_MATCH` | **True** | 启用向量匹配补充精确匹配 | 精确匹配: 1.0<br>向量匹配: similarity² |
| `SEARCH_LAMBDA` | **1e-9** | 搜索阶段衰减参数（已弃用） | 现使用统一的 DECAY_RATE |

## 二、多维度事件合并算法详解

### 1. 算法原理

LiMem 采用**四维度加权相似度算法**进行事件合并，替代单一的向量相似度匹配。

**核心公式**：
```
综合相似度 = α × 语义相似度
           + β × 实体重叠度
           + γ × 时间接近度
           + δ × 动作匹配度
```

### 2. 四个维度详解

#### 2.1 语义相似度（Semantic Similarity）

**计算方法**：余弦相似度（Cosine Similarity）
```python
semantic_sim = cosine_similarity(event_embedding, stored_embedding)
# 范围：0.0 - 1.0
```

**物理意义**：衡量两个事件在语义空间中的接近程度

**示例**：
```
事件A: "用户请求播放音乐，车机播放了还是分开（张叶蕾）"
事件B: "用户请求播放歌曲，车机回应已正在播放"
语义相似度: 0.92 (高度相似)
```

#### 2.2 实体重叠度（Entity Overlap）

**计算方法**：Jaccard 相似度
```python
intersection = len(实体集合A & 实体集合B)
union = len(实体集合A | 实体集合B)
entity_sim = intersection / union
# 范围：0.0 - 1.0
```

**物理意义**：衡量两个事件共享实体的比例

**示例**：
```
事件A实体: ['用户', '张叶蕾', '音乐', '车机', '还是分开']
事件B实体: ['用户', '车机', '歌', '歌曲名', '歌手']
共同实体: ['用户', '车机']
实体重叠度: 2/6 = 0.33
```

#### 2.3 时间接近度（Temporal Proximity）

**计算方法**：指数衰减
```python
time_diff = abs(current_time - last_active)
if time_diff <= MERGE_TIME_WINDOW:  # 默认 300 秒
    time_sim = exp(-3.0 * time_diff / MERGE_TIME_WINDOW)
else:
    time_sim = 0.0
# 范围：0.0 - 1.0
```

**物理意义**：衡量两个事件在时间上的接近程度

**时间衰减曲线**：
```
时间差（秒）  时间相似度
━━━━━━━━━━━━━━━━━━━━━━
0           1.000  ████████████
30          0.741  █████████▏
60          0.549  ██████▏
120         0.301  ███▏
180         0.165  ██▏
240         0.091  █▏
300         0.050  ▏（边界）
>300        0.000  （超出窗口）
```

#### 2.4 动作匹配度（Action Match）

**计算方法**：完全匹配
```python
action_sim = 1.0 if incoming_action == stored_action else 0.0
# 范围：0 或 1
```

**物理意义**：衡量两个事件的动作类型是否相同

**示例**：
```
事件A动作: "播放"
事件B动作: "播放"
动作匹配度: 1.0

事件A动作: "播放"
事件B动作: "导航"
动作匹配度: 0.0
```

### 3. 权重配置策略

#### 3.1 默认配置（推荐）
```bash
MERGE_WEIGHT_SEMANTIC=0.4  # 40%
MERGE_WEIGHT_ENTITY=0.3    # 30%
MERGE_WEIGHT_TIME=0.2      # 20%
MERGE_WEIGHT_ACTION=0.1    # 10%
```

**适用场景**：通用场景，平衡语义和实体

#### 3.2 语义驱动配置
```bash
MERGE_WEIGHT_SEMANTIC=0.5  # 50%
MERGE_WEIGHT_ENTITY=0.3    # 30%
MERGE_WEIGHT_TIME=0.1      # 10%
MERGE_WEIGHT_ACTION=0.1    # 10%
```

**适用场景**：
- 事件表述差异大但语义相似
- 需要识别同义表述
- LLM 生成的摘要不稳定

#### 3.3 实体驱动配置
```bash
MERGE_WEIGHT_SEMANTIC=0.3  # 30%
MERGE_WEIGHT_ENTITY=0.5    # 50%
MERGE_WEIGHT_TIME=0.1      # 10%
MERGE_WEIGHT_ACTION=0.1    # 10%
```

**适用场景**：
- 实体是区分事件的关键
- 需要精确匹配人名、地名、媒体名
- 避免语义相似但实体不同的事件合并

#### 3.4 时间敏感配置
```bash
MERGE_WEIGHT_SEMANTIC=0.3  # 30%
MERGE_WEIGHT_ENTITY=0.2    # 20%
MERGE_WEIGHT_TIME=0.4      # 40%
MERGE_WEIGHT_ACTION=0.1    # 10%
```

**适用场景**：
- 时间是区分事件的关键因素
- 同一场景在不同时间发生
- 需要区分"连续会话"和"独立事件"

### 4. 实际案例对比

#### 案例 1：音乐播放事件

**场景**：两个相似的播放请求
```
Episode 1: "给我播放首音乐"
  → Event: "用户请求播放音乐，车机播放了还是分开（张叶蕾）"
  → 实体: ['用户', '张叶蕾', '音乐', '车机']

Episode 2: "给我放首歌"
  → Event: "用户请求播放歌曲，车机回应已正在播放"
  → 实体: ['用户', '车机', '歌']
```

**旧算法（纯向量）**：
```
语义相似度: 0.92
综合评分: 0.92
阈值: 0.70
结果: ❌ 0.92 > 0.70，但仍然分开（因为时间差或其他因素）
```

**新算法（多维度）**：
```
语义相似度: 0.92
实体重叠度: 0.50 (2/4 共同实体)
时间接近度: 1.00 (同一时间段)
动作匹配度: 1.00 (都是"播放")
综合评分: 0.4×0.92 + 0.3×0.50 + 0.2×1.00 + 0.1×1.00 = 0.818
阈值: 0.65
结果: ✅ 0.818 > 0.65，成功合并
```

#### 案例 2：相同时间不同事件

**场景**：同一时间的不同操作
```
Episode 1: "播放音乐"
  → Event: "用户请求播放音乐"
  → 实体: ['用户', '音乐']

Episode 2: "导航到长城"
  → Event: "用户请求导航到长城"
  → 实体: ['用户', '长城']
```

**旧算法**：
```
语义相似度: 0.45
综合评分: 0.45
结果: ✅ 正确分开
```

**新算法**：
```
语义相似度: 0.45
实体重叠度: 0.33 (1/3 共同实体)
时间接近度: 1.00 (同一时间)
动作匹配度: 0.00 (播放 vs 导航)
综合评分: 0.4×0.45 + 0.3×0.33 + 0.2×1.00 + 0.1×0.00 = 0.479
结果: ✅ 0.479 < 0.65，正确分开
```

## 四、检索参数调优指南

### 1. TOP_K 选择策略

| TOP_K | 上下文长度 | 答案质量 | 响应速度 | 适用场景 |
|-------|-----------|----------|----------|----------|
| 3 | 短 | 精确但可能不完整 | 快 | 简单查询 |
| **5** | **适中** | **平衡** | **适中** | ✅ **通用场景** |
| 10 | 长 | 全面但可能噪音多 | 慢 | 复杂查询 |
| 15+ | 很长 | 噪音增加 | 很慢 | 需要大量上下文 |

### 2. 实体提取数量

| MAX_ENTITIES | 查询理解 | 检索范围 | 噪音水平 | 推荐 |
|--------------|----------|----------|----------|------|
| 5 | 核心实体 | 窄 | 低 | 简单查询 |
| **10** | **关键实体** | **适中** | **中** | ✅ **通用** |
| 15 | 全面实体 | 宽 | 高 | 复杂查询 |
| 20+ | 过多实体 | 过宽 | 很高 | ❌ 不推荐 |

### 3. 向量匹配参数

**SEARCH_VECTOR_THRESHOLD 权衡**：

```
阈值 = 0.3 (低阈值)
━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 高召回率（找到更多相关实体）
❌ 低精确率（引入噪音实体）
适用：需要高召回的场景

阈值 = 0.5 (推荐值)
━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 平衡召回率和精确率
✅ 过滤掉不相关的实体
适用：通用场景

阈值 = 0.7 (高阈值)
━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 高精确率（只保留高度相关实体）
❌ 低召回率（可能漏掉相关实体）
适用：需要高精度的场景
```

## 七、常见问题与解决方案

### Q1: 记忆检索不准确

**症状**：查询返回不相关的事件

**诊断**：
1. 检查 `SIMILARITY_THRESHOLD` 是否过低（< 0.60）
2. 检查 `SEARCH_VECTOR_THRESHOLD` 是否过低（< 0.4）
3. 检查实体提取是否准确

**解决方案**：
```bash
# 提高阈值，减少噪音
SIMILARITY_THRESHOLD=0.75
SEARCH_VECTOR_THRESHOLD=0.6
```

### Q2: 记忆过度碎片化

**症状**：同一事件被拆分成多个小事件

**诊断**：
1. 检查 `SIMILARITY_THRESHOLD` 是否过高（> 0.80）
2. 检查 Prompt 是否过于细化

**解决方案**：
```bash
# 降低阈值，允许合并
SIMILARITY_THRESHOLD=0.65
```

### Q3: 老记忆无法检索

**症状**：几个月前的记忆检索不到

**诊断**：
1. 检查 `DECAY_RATE` 是否过大（> 1e-7）
2. 计算半衰期是否过短

**解决方案**：
```bash
# 降低衰减率，延长记忆
DECAY_RATE=1e-9  # 半衰期 22 年
```

### Q4: 检索速度慢

**症状**：查询响应时间 > 2 秒

**诊断**：
1. 检查 `SEARCH_TOP_K` 是否过大（> 10）
2. 检查 `SEARCH_VECTOR_TOP_K` 是否过大（> 15）
3. 检查数据库索引

**解决方案**：
```bash
# 减少检索数量
SEARCH_TOP_K=5
SEARCH_VECTOR_TOP_K=8
```
